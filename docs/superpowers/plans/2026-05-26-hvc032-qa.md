### Task 1: Fix DCOM PID Wire Mismatch

**Files:**
- Modify: `payloads/Demon/src/commands/Command_Lateral.c` (function `LateralDcomExec`, near line 576)

**Problem:** `LateralDcomExec` writes only `Success` (1 × int32) into the package. The failure-path early-returns in `CommandLateral` write `Success + PID` (2 × int32), and the Go `TaskDispatch` parser expects exactly `CanIRead(ReadInt32, ReadInt32)` after `SubCommand`. DCOM responses always hit the "invalid response" branch because the second int32 is missing.

- [ ] **Step 1: Locate the single-int write in LateralDcomExec**

```bash
grep -n "PackageAddInt32.*Success\|PackageAddInt32.*Pid" \
    payloads/Demon/src/commands/Command_Lateral.c
```

Expected output contains two lines for `LateralWmiExec` (Success + Pid) and **one line** for `LateralDcomExec` (Success only, around line 576).

- [ ] **Step 2: Add the missing PID field**

In `payloads/Demon/src/commands/Command_Lateral.c`, after line `PackageAddInt32( Package, (UINT32) Success );` inside `LateralDcomExec` (at the DCOM_OUT cleanup block, before the closing `}`), add:

```c
    PackageAddInt32( Package, (UINT32) Success );
    PackageAddInt32( Package, 0 );   /* DCOM has no process PID to report */
```

Replace the existing single line:
```c
    PackageAddInt32( Package, (UINT32) Success );
```
with:
```c
    PackageAddInt32( Package, (UINT32) Success );
    PackageAddInt32( Package, 0 );   /* DCOM - no process PID */
```

- [ ] **Step 3: Verify the fix**

```bash
grep -n "PackageAddInt32.*Success\|PackageAddInt32.*0.*PID\|PackageAddInt32.*0.*DCOM\|PackageAddInt32.*0.*process" \
    payloads/Demon/src/commands/Command_Lateral.c
```

Both `LateralWmiExec` and `LateralDcomExec` now write exactly 2 int32 fields (Success + PID/0).

- [ ] **Step 4: Commit**

```bash
git add payloads/Demon/src/commands/Command_Lateral.c
git commit -m "fix(lateral): DCOM response missing PID field - add 0 sentinel"
```

### Task 2: Fix Mqi.pItf Potential Leak in LateralDcomExec

**Files:**
- Modify: `payloads/Demon/src/commands/Command_Lateral.c` (function `LateralDcomExec`, near line 509)

**Problem:** If `CoCreateInstanceEx` fails (`FAILED(Hr)` is true) but still wrote a non-NULL interface pointer into `Mqi.pItf` (COM partial-QI scenario), the code jumps to `DCOM_OUT` before `pDisp = Mqi.pItf` executes, leaving `pDisp = NULL`. `ComRelease( &pDisp )` at cleanup is then a no-op and `Mqi.pItf` leaks.

- [ ] **Step 1: Locate the CoCreateInstanceEx failure branch**

```bash
grep -n "Mqi.pItf\|DCOM_OUT\|goto DCOM_OUT" \
    payloads/Demon/src/commands/Command_Lateral.c | head -15
```

Expected: the `if ( FAILED(Hr) || FAILED(Mqi.hr) || !Mqi.pItf )` guard at line ~509 followed by `goto DCOM_OUT`.

- [ ] **Step 2: Add release of Mqi.pItf on the failure-goto path**

Replace the existing guard block:
```c
    if ( FAILED( Hr ) || FAILED( Mqi.hr ) || !Mqi.pItf ) {
        PRINTF( "LateralDcomExec CoCreateInstanceEx failed: outer=0x%x qi=0x%x\n", Hr, Mqi.hr )
        goto DCOM_OUT;
    }
    pDisp = Mqi.pItf;
```
with:
```c
    if ( FAILED( Hr ) || FAILED( Mqi.hr ) || !Mqi.pItf ) {
        PRINTF( "LateralDcomExec CoCreateInstanceEx failed: outer=0x%x qi=0x%x\n", Hr, Mqi.hr )
        if ( Mqi.pItf ) { ComRelease( &Mqi.pItf ); }   /* partial QI result */
        goto DCOM_OUT;
    }
    pDisp = Mqi.pItf;
```

- [ ] **Step 3: Verify**

```bash
grep -n "Mqi.pItf" payloads/Demon/src/commands/Command_Lateral.c
```

Expected: the guard block now has `ComRelease( &Mqi.pItf )` before `goto DCOM_OUT`.

- [ ] **Step 4: Commit**

```bash
git add payloads/Demon/src/commands/Command_Lateral.c
git commit -m "fix(lateral): release Mqi.pItf on partial CoCreateInstanceEx failure"
```

- [ ] **Step 3: Verify NETINFO ADAPTERS response sentinel**

C sends per-adapter: `[Name(str), Desc(str), IP(str), Mask(str), GW(str), MAC(bytes)]` × N adapters, then `0(int32)` sentinel.

Go parser detects end via:
```go
for Parser.CanIRead([]parser.ReadType{parser.ReadBytes}) {
    Name := Parser.ParseString()
    if Name == "" { break }
    ...
}
if Parser.CanIRead([]parser.ReadType{parser.ReadInt32}) {
    Parser.ParseInt32()  // consume sentinel
}
```

Verify the sentinel is an `int32` (4 bytes) not a zero-length string. Check C side:
```bash
grep -n "PackageAddInt32.*0\|PackageAddString\|PackageAddBytes" \
    payloads/Demon/src/commands/Command_Netinfo.c | grep -A2 "sentinel\|end-of"
```

Expected: `PackageAddInt32( Package, 0 )` is the list terminator, not `PackageAddString( Package, "" )`.

**Note:** The Go parser loop uses `CanIRead(ReadBytes)` which reads a length-prefixed field. When the sentinel `int32(0)` arrives, `CanIRead(ReadBytes)` will read 4 bytes and interpret them as a zero-length string (length=0). The loop then calls `Parser.ParseString()` which returns `""` and `break` fires. This is fragile — the sentinel is effectively consumed by the loop, not the explicit `ParseInt32()` call. Verify this works correctly by tracing the parser logic.

- [ ] **Step 4: Verify NETINFO ARP response sentinel**

C sends per-row: `[IfIdx(int32), Family(int32), IP(bytes), MAC(bytes), State(int32)]` × N rows, then `0(int32)` sentinel.

Go reads:
```go
for Parser.CanIRead([]parser.ReadType{parser.ReadInt32}) {
    IfIdx := Parser.ParseInt32()
    if IfIdx == 0 { break }
    ...
}
```

This is clean. Verify the sentinel `0(int32)` does NOT get consumed as the start of a valid row before the loop checks `IfIdx == 0`. Expected: the loop reads `IfIdx=0` and breaks correctly.
